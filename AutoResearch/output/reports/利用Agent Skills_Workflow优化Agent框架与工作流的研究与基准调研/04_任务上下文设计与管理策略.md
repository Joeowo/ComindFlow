## 任务上下文设计与管理策略：优化 Agent 框架与工作流的研究与基准调研

### 1. 研究背景与核心问题
随着大语言模型（LLM）驱动的自主 Agent 走向复杂的多步任务执行，**任务上下文的设计与管理**成为决定 Agent 上限的关键因素。上下文不仅指当前对话历史，更涵盖长期记忆、子目标状态、工具调用结果、多智能体共享信息以及从过往经验中提炼的技能。其核心挑战包括：

- **有限上下文窗口**：即使支持 128k–1M token，长期复杂任务仍易导致关键信息丢失或成本失控。
- **上下文污染与遗忘**：无关或过时信息干扰推理，遗漏早期约束或中间结果。
- **跨会话/跨任务的上下文迁移**：如何在多次会话、多个技能或不同工作流中复用结构化上下文。
- **动态上下文的编排**：在反思、重规划、多代理协作中高效选择、压缩和传递上下文。

下文将从**设计策略、管理策略、技能/工作流优化、基准评估和代表性框架**五个层面，对当前研究进行系统梳理。

### 2. 任务上下文设计策略：结构、压缩与分解

#### 2.1 记忆增强的上下文架构
将 Agent 的上下文扩展为一套可读写的记忆系统，使模型能够“回忆”跨轮次的信息。

| 方法 | 核心机制 | 适用场景 |
|------|----------|----------|
| **MemGPT** | 虚拟上下文分页：将内存分为主上下文（可写入）和外部存储（可检索），通过中断实现上下文调度 | 长对话、文档分析、长期用户记忆 |
| **Generative Agents** | 记忆流（Memory Stream）+ 反思（Reflection）+ 检索加权（Recency, Importance, Relevance） | 沙盒模拟、社会行为建模 |
| **MemoryBank** | 基于向量的持久记忆库，支持新增、遗忘、检索和摘要更新 | 个人助理、长期知识积累 |
| **ChatDev** | 以聊天链（chat chain）为载体，维持多阶段的交流历史作为共享上下文 | 软件协作开发 |
| **LongMem** | 在 Transformer 内部加入持久记忆模块，实现跨输入的长程依赖 | 长文本理解、终身学习 |

#### 2.2 上下文压缩与精炼
在不扩大上下文窗口的前提下，压缩冗余信息以保留关键内容。

- **LLMLingua 系列**：
  - **LLMLingua**：利用小型模型对提示进行压缩，保持语言模型的输出一致性。
    - 来源: arXiv:2310.05736 (2023) — Huiqiang Jiang et al.
    - 链接: https://arxiv.org/abs/2310.05736
  - **LongLLMLingua**：针对长上下文场景，通过粗到精的压缩策略和文档重排序增强。
    - 来源: arXiv:2310.06839 (2023) — Huiqiang Jiang et al.
    - 链接: https://arxiv.org/abs/2310.06839
  - **LLMLingua-2**：任务无关的数据蒸馏压缩，提升压缩率和效率。
    - 来源: arXiv:2403.12968 (2024) — Zhuoshi Pan et al.
    - 链接: https://arxiv.org/abs/2403.12968

- **Gist Tokens**：学习少量“要旨” token 来取代整个提示，实现极低成本的上下文压缩。
  - 来源: arXiv:2304.08467 (2023) — Jesse Mu et al.
  - 链接: https://arxiv.org/abs/2304.08467

- **Selective Context**：基于信息论选择最相关上下文段落（常作为一种实用技巧，但缺乏正式论文）。部分框架（如 LlamaIndex）实现了上下文窗口的重排序与压缩。

#### 2.3 层次化任务规划与上下文分解
通过将复杂任务拆解为多个子任务，降低单个推理步骤的上下文负载。

| 方法 | 分解方式 | 上下文优势 |
|------|----------|------------|
| **Least-to-Most Prompting** | 顺序分解：先解决子问题，再将答案纳入后续上下文 | 逐步积累正确答案，防止问题漂移 |
| **Plan-and-Solve** | 首先生成全局计划，再逐步执行 | 计划作为上下文骨架，指导执行 |
| **Tree of Thoughts (ToT)** | 树状搜索多个推理路径 | 维护多条上下文分支，支持回溯 |
| **DEPS** | 描述、解释、规划、选择交互式规划 | 在开放世界环境多任务中整合视觉与语义上下文 |
| **LLM+P** | 利用经典规划器（PDDL）生成最优计划 | 将自然语言上下文转化为形式化规划问题，保证计划可行性 |

#### 2.4 技能库与模块化上下文注入
将经过验证的执行过程封装为可复用的**技能（skill）**，在需要时将技能描述和实现代码作为上下文注入，避免重复推理。

- **Voyager**：在 Minecraft 中通过代码生成、执行和迭代改进构建**技能库**。每个技能是一个可执行 JavaScript 函数，Agent 通过检索相关技能并将它们纳入提示中解决新任务。
  - 来源: arXiv:2305.16291 (2023) — Guanzhi Wang et al.
  - 链接: https://arxiv.org/abs/2305.16291
  - 核心贡献：技能作为长期记忆实体，通过上下文注入实现终生学习。

- **Toolformer**：通过自监督学习教会模型自主生成工具调用命令，工具 API 的描述构成按需上下文。
  - 来源: arXiv:2302.04761 (2023) — Timo Schick et al.
  - 链接: https://arxiv.org/abs/2302.04761

- **AutoAct**：从零开始自动学习任务解决技能，以“子智能体”上下文组的形式组织，实现上下文高效迁移。
  - 来源: arXiv:2403.06968 (2024) — Ke Li et al.
  - 链接: https://arxiv.org/abs/2403.06968

### 3. 工作流中的上下文管理策略

#### 3.1 动态上下文选择与反思
Agent 在执行过程中根据中间结果主动剪枝、补充或重写上下文。

- **Reflexion**：引入“口头强化学习”，将历史反馈和失败记忆存储为长期上下文，在后续尝试中根据检索结果注入提示。
  - 来源: arXiv:2303.11366 (2023) — Noah Shinn et al.
  - 链接: https://arxiv.org/abs/2303.11366

- **Self-Refine**：迭代式自我反馈与修正，每一次迭代将前一轮的输出和反馈作为上下文继续精炼。
  - 来源: arXiv:2303.17651 (2023) — Aman Madaan et al.
  - 链接: https://arxiv.org/abs/2303.17651

- **AdaPlanner**：根据环境反馈自适应调整计划，并在计划上下文中保留成功/失败状态以影响后续生成。
  - 来源: arXiv:2305.16653 (2023) — Haotian Sun et al.
  - 链接: https://arxiv.org/abs/2305.16653

#### 3.2 多智能体上下文传递与状态管理
多个智能体协作时，上下文需要在智能体间高效传递、合并与同步。

| 框架/方法 | 上下文传递机制 | 特点 |
|-----------|----------------|------|
| **AutoGen** | 对话驱动：智能体间通过多轮消息（文本、代码、函数调用）交换上下文 | 支持动态群组对话、人类介入和状态转换 |
| **MetaGPT** | 结构化文档流转：需求、设计、测试等文档作为共享上下文 | 模拟软件公司 SOP，减少对话噪音 |
| **CAMEL** | 角色扮演通信：用户代理和助手代理持续交换消息 | 通过“起始提示”设定全局任务上下文 |
| **ProAgent** | 构建工作流图：利用代码生成构造 DAG，数据依赖关系作为上下文传递依据 | 从 RPA 升级到智能过程自动化，上下文路径清晰 |

#### 3.3 执行状态跟踪与代码上下文
对于代码生成和交互式任务，执行环境和变量状态本身构成了关键上下文。

- **TaskWeaver**：以代码优先的 Agent 框架，将用户请求转化为可执行代码，整个执行过程中的变量和状态被持久化，供后续步骤引用。

### 4. 基准测试与评估体系

#### 4.1 Agent 通用能力基准（含上下文相关维度）
这些基准通过长序列、多步任务间接评估上下文管理能力。

- **AgentBench**：涵盖 OS 交互、数据库、知识图谱、Web 操作等 8 个环境，测试长任务中 Agent 的规划与记忆。
  - 来源: arXiv:2308.03688 (2023) — Xiao Liu et al.
  - 链接: https://arxiv.org/abs/2308.03688

- **WebArena**：搭建真实站点模拟，要求 Agent 在复杂网页中完成多步操作，考验上下文捕捉与状态跟踪。
  - 来源: arXiv:2307.13854 (2023) — Shuyan Zhou et al.
  - 链接: https://arxiv.org/abs/2307.13854

- **GAIA**：针对类人助理的基准，设计需推理、多模态信息融合的难问题，避免简单工具调用可解。
  - 来源: arXiv:2311.12983 (2023) — Grégoire Mialon et al.
  - 链接: https://arxiv.org/abs/2311.12983

- **SWE-bench**：基于 GitHub 真实 issue，要求 Agent 理解代码仓库上下文并生成补丁。

  - 链接: https://arxiv.org/abs/2310.06770

### 6. 未来研究趋势与开放挑战

1. **上下文预算分配与调度**：如何在有限注意力下动态分配 token 比例给指令、记忆、工具结果和规划分支，目前仍缺乏理论模型。
2. **跨会话上下文的持续学习**：Agent 在成千上万次交互后累积的“经验”，如何安全地更新而避免混乱或遗忘，MemGPT 和 Voyager 提供了初步方案，但尚未与人类反馈对齐。
3. **层次化技能图谱的自动构建**：如果技能之间的关系被建模为图谱（前提条件、后置效果、数据流），Agent 可以自动组合技能并仅加载相关子图作为上下文，减少盲目检索。
4. **上下文质量的自动评估与修复**：当检测到矛盾或不准确时，Agent 应主动重写部分上下文或触发反思，类似 AdaPlanner 的自适应机制需进一步发展。
5. **隐私与安全**：长期上下文持久化带来了用户敏感信息遗忘和注入攻击的风险，需要上下文隔离、差分隐私或访问控制策略。
6. **基准的本土化**：现有基准多为合成或英文，构建面向长上下文、多语言、多模态和真实工业场景的上下文管理基准是迫切需求。
