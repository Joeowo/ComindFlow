### 2. 自动化测试生成

#### 2.1 符号执行

**经典工作：KLEE**

- **来源**: OSDI 2008, 引用超过 3000 次
- **作者**: Cristian Cadar, Daniel Dunbar, Dawson Engler
- **标题**: *KLEE: Unassisted and Automatic Generation of High-Coverage Tests for Complex Systems Programs*
- **链接**: [https://www.doc.ic.ac.uk/~cristic/papers/klee-osdi-08.pdf](https://www.doc.ic.ac.uk/~cristic/papers/klee-osdi-08.pdf)
- **核心贡献**: 提出基于 LLVM 的符号执行引擎，能够自动为 C 程序生成高覆盖率测试用例，并发现大量真实缺陷。

**近期进展：SymQEMU**

- **来源**: arXiv:2203.14882 (2022)
- **作者**: Seulbae Kim et al.
- **链接**: [https://arxiv.org/abs/2203.14882](https://arxiv.org/abs/2203.14882)
- **核心贡献**: 基于 QEMU 的二进制符号执行框架，支持多种架构，无需源码即可生成测试用例，显著降低二进制测试门槛。

#### 2.2 搜索式软件测试 (SBST)

**代表性工具：EvoSuite**

- **来源**: ESEC/FSE 2011, 演化计算领域标志性工作
- **作者**: Gordon Fraser, Andrea Arcuri
- **标题**: *EvoSuite: Automatic Test Suite Generation for Object-Oriented Software*
- **链接**: [https://www.evosuite.org](https://www.evosuite.org) (官网) / 论文: [https://doi.org/10.1145/2025113.2025179](https://doi.org/10.1145/2025113.2025179)
- **核心贡献**: 使用遗传算法为 Java 类生成 JUnit 测试套件，能够同时优化分支覆盖率和测试套件规模。

**近期扩展：Whole-Suite Approach**

- **来源**: IEEE Transactions on Software Engineering, 2015
- **作者**: Gordon Fraser, Andrea Arcuri
- **链接**: [https://arxiv.org/abs/1408.1009](https://arxiv.org/abs/1408.1009) (预印本)
- **核心贡献**: 提出“全套件”适应度函数，在演化过程中直接优化整个测试套件的覆盖，优于传统逐个测试生成方法。

### 3. 模糊测试 (Fuzzing)

模糊测试是当前漏洞发现和测试生成最活跃的领域之一。

#### 3.1 引导式模糊测试

**AFL (American Fuzzy Lop)**

- **作者**: Michal Zalewski, Google, 2013 (首次发布)
- **官网/仓库**: [https://lcamtuf.coredump.cx/afl/](https://lcamtuf.coredump.cx/afl/)
- **核心特性**: 使用编译时插桩和边覆盖反馈引导变异，是覆盖率引导模糊测试的里程碑，衍生出大量后续工作。

**AFL++ (增强版)**

- **GitHub**: [https://github.com/AFLplusplus/AFLplusplus](https://github.com/AFLplusplus/AFLplusplus)
- **核心特性**: 融合多种插桩技术、变异策略和调度算法，支持 QEMU、Unicorn 等多种模式，是目前最活跃的社区模糊测试框架。

**LibFuzzer**

- **来源**: LLVM 项目自带, [https://llvm.org/docs/LibFuzzer.html](https://llvm.org/docs/LibFuzzer.html)
- **核心特性**: 进程内、基于覆盖率的模糊测试引擎，与 Sanitizers 紧密集成，适合库、API 测试。

#### 3.2 混合模糊测试 (Hybrid Fuzzing)

**QSYM**

- **来源**: USENIX Security 2018
- **作者**: Insu Yun et al.
- **标题**: *QSYM: A Practical Concolic Execution Engine Tailored for Hybrid Fuzzing*
- **链接**: [https://www.usenix.org/conference/usenixsecurity18/presentation/yun](https://www.usenix.org/conference/usenixsecurity18/presentation/yun)
- **核心贡献**: 将符号执行与模糊测试深度集成，使用指令级别符号模型，解决了传统混合模糊测试的性能瓶颈。

**Sydr**

- **来源**: arXiv:2302.00784 (2023)
- **作者**: Institute for System Programming, RAS
- **链接**: [https://arxiv.org/abs/2302.00784](https://arxiv.org/abs/2302.00784)
- **核心贡献**: 动态符号执行与模糊测试混合框架，支持 x86/x64/ARM，提供路径谓词的精简和缓存机制，有效提升漏洞发现效率。

#### 3.3 基于AI的模糊测试

**NEUZZ**

- **来源**: arXiv:1907.05620 (2019)
- **作者**: Dongdong She et al.
- **链接**: [https://arxiv.org/abs/1907.05620](https://arxiv.org/abs/1907.05620)
- **核心贡献**: 使用梯度提升决策树代理程序分支行为，通过神经网络平滑梯度实现高效变异，开创神经程序平滑模糊测试方向。

**MTFuzz**

- **来源**: arXiv:2205.10493 (2022)
- **作者**: Chenyang Lyu et al.
- **链接**: [https://arxiv.org/abs/2205.10493](https://arxiv.org/abs/2205.10493)
- **核心贡献**: 多任务神经网络模糊测试，通过共享表示和任务特定头，同时预测多个覆盖目标，提高模糊测试的探索效率。

### 4. 基于AI/机器学习的测试

#### 4.1 测试预言生成

- **论文**: *DeepTest: Automated Testing of Deep-Neural-Network-Driven Autonomous Cars*
- **来源**: ICSE 2018, arXiv:1708.08559
- **作者**: Yuchi Tian et al.
- **链接**: [https://arxiv.org/abs/1708.08559](https://arxiv.org/abs/1708.08559)
- **核心贡献**: 利用图像变换（雨、雾、亮度）生成变异数据，测试自动驾驶 DNN 的鲁棒性，是 DNN 测试的早期里程碑。

#### 4.2 基于大语言模型 (LLM) 的测试生成

**TestPilot**

- **来源**: arXiv:2304.07587 (2023)
- **作者**: Max Schäfer et al.
- **链接**: [https://arxiv.org/abs/2304.07587](https://arxiv.org/abs/2304.07587)
- **核心贡献**: 使用 LLM (Codex) 为 JavaScript 库生成单元测试，无需训练，通过提示工程即可达到可用性，开创 LLM 用于测试生成的先河。

**CodaMosa**

- **来源**: arXiv:2310.14646 (2023)
- **作者**: Caroline Lemieux et al.
- **链接**: [https://arxiv.org/abs/2310.14646](https://arxiv.org/abs/2310.14646)
- **核心贡献**: 将 LLM 生成的测试与 SBST (EvoSuite) 结合，利用 LLM 提供语义上有意义的种子，大幅提升搜索式测试的效率。

- **核心贡献**: 形式化 TCP 问题，系统比较了覆盖、来源、修改等优先排序策略，是 TCP 领域的奠基之作。

**基于预测模型的优先排序**

- **来源**: arXiv:2301.00439 (2023)
- **作者**: Matteo Biagiola et al.

- **链接**: [https://arxiv.org/abs/2301.00439](https://arxiv.org/abs/2301.00439)
- **核心贡献**: 系统性评估多种 ML 模型用于 TCP 的有效性，发现随机森林通常优于其他模型，且特征选择对结果影响巨大。

- **来源**: FSE 2014
- **作者**: Milos Gligoric et al.
- **标题**: *Ekstazi: Lightweight Test Selection*

### 7. 当前挑战与未来趋势

#### 7.1 挑战

- **测试预言问题**：自动判断测试结果正确性依然困难，尤其在复杂系统和无规范场景中。
- **脆弱测试 (Flaky Tests)**：非确定性的测试失败严重降低 CI/CD 可信度，根因多样（异步、时间依赖等）。
- **真实复杂软件的自动化测试**：容器、微服务、分布式系统中的状态和非确定性使得端到端测试自动化困难。
- **安全关键系统的测试完备性**：难以量化“足够”的测试量，形式化验证在其中回归缓慢。

#### 7.2 趋势

1. **LLM 与测试深度融合**：不仅生成测试代码，还能生成测试预言、分析失败原因、修复 flaky 测试（参见 TestPilot, CodaMosa）。
2. **基于属性的测试 (Property-Based Testing) 复兴**：如 Python 的 Hypothesis、Erlang 的 QuickCheck，快速生成满足特定属性的输入。
3. **联邦模糊测试 (Collaborative Fuzzing)**：多引擎、多实例并行协作，如 ClusterFuzz、OneFuzz（微软），利用云资源进行大规模持续模糊测试。
4. **测试环境的模拟与录制回放**：如 Speedscope、rr 等项目，通过记录和重放系统调用辅助调试和测试。
5. **安全左移与 DevSecOps**：在开发生命周期早期集成安全测试 (SAST, DAST, IAST)，工具链越来越自动化。

---

### 8. 总结

软件测试是一个纵横学术与工业的活跃领域，覆盖从传统白盒测试到前沿的基于 AI 的漏洞发现。模糊测试和基于搜索的测试生成仍然是主流的自动漏洞与 bug 发现手段，而 LLM 的介入正在快速改变测试生成的范式。未来，测试工具将更加智能（自主生成、自主修复）和无缝集成到开发流程中。
